# GitHub PR Lane Workflow

Use this skill when coordinating development work across GitHub issues and pull requests with implementation agents, independent reviewers, CI gates, and backlog selection.

This workflow is for keeping one active delivery lane moving without losing safety discipline. It works best when GitHub is the source of truth and an agent can read issues, PRs, reviews, CI status, branches, and local repo state.

## Core Loop

1. Identify the active lane from GitHub, not memory.
2. Read the active PR, linked issue, latest commits, review comments, and CI/check status.
3. Prioritize blocking work before new work.
4. Delegate implementation or review-fix work to the implementation agent.
5. Run the exact relevant local gates before every push.
6. Push fixes only after local verification passes.
7. Start or refresh an independent review for the exact current head SHA.
8. Poll CI and review together.
9. Fix blocking findings immediately.
10. Merge only when all merge gates are satisfied.
11. Run a short delivery retro.
12. Review the backlog, batch related issues, and start the next lane.

## Priority Order

Always handle work in this order:

1. Active PR with blocking review feedback.
2. Active PR with failing CI.
3. Active PR awaiting independent review.
4. Green and approved PR ready to merge.
5. Backlog planning pass across recent merged PR comments and all open GitHub issues.

Do not start new implementation work while a current PR has blocking review feedback, failing required checks, unresolved merge conflicts, or missing required independent review.

Background backlog planning is allowed while a PR is waiting on CI or review, but implementation for the next lane must wait until the current lane has no actionable blocker and the next work is clearly parallel-safe.

## Roles

Adapt the names to the local team, but preserve the separation of duties:

- Coordinator: owns GitHub state, lane priority, delegation, merge gates, and status updates.
- Implementation agent: pulls issues, implements changes, and handles review-fix work.
- Independent reviewer: reviews the exact current PR head for blocking findings.
- Supplemental reviewer: optional browser or second-agent review for early signals.
- Human owner: only needed for product, architecture, safety, credential, spending, or unrecoverable blocker decisions.

Do not ask the human for ordinary implementation details, normal test failures, or routine review feedback.

## Active PR Triage

For the active PR, inspect:

- PR title, body, linked issue, branch, author, and mergeability.
- Latest head SHA.
- Required checks and CI failures.
- Open review threads and unresolved comments.
- Recent commits and changed files.
- Whether any reviewer approval or no-blocking finding applies to the latest head.

Treat review comments on old commits as unresolved until confirmed fixed or marked obsolete with evidence.

## Review-Fix Interrupts

New blocking review feedback is an interrupt.

When a blocker appears:

1. Start implementation or review-fix work immediately.
2. Give the worker the PR URL, exact head SHA, linked issue, acceptance criteria, blocker text, and relevant file references.
3. Require focused fixes without unrelated refactors.
4. Run local gates matching the risk surface.
5. Push the fix.
6. Start independent review again for the new exact head SHA.

Do not send a status-only update about blockers unless a fix worker has already started or a real blocker prevents starting one.

## Local Gates Before Push

Before every push, run the exact relevant local gate instead of a partial approximation:

- Formatting or format-check.
- Lint.
- Typecheck when relevant.
- Tests matching the CI risk surface.
- Full test suite when shared helpers, scheduling, persistence, clock/session logic, durable execution, safety boundaries, migrations, or previously CI-only failures are touched.

Focused tests are useful for inner-loop development, but they do not replace the appropriate pre-push gate.

## Independent Review

After every pushed code or docs change on an active PR:

1. Identify the exact current head SHA.
2. Start independent review for that SHA as soon as CI is green or the diff is reviewable.
3. Comment on the PR that review is actively running for that exact SHA.
4. Ask for blocking findings only, with file and line references.
5. If no blockers are found, attach an explicit no-blocking finding or approval to the PR for that exact SHA.
6. If findings are found, attach them to the PR and start review-fix work.

If review does not return promptly, verify it targets the latest head and start a fallback independent review rather than waiting indefinitely.

A requested review is not enough to merge. The current PR head needs an explicit approval or no-blocking finding attached to the PR.

## CI and Review Polling

Poll CI and independent review together instead of serializing them.

On each poll:

- If CI failed because of formatting or lint only, repair automatically, push, and restart review for the new head.
- If CI failed nontrivially, delegate or fix based on the failure surface.
- If review found blockers, interrupt into review-fix work.
- If review is stale for an old head, restart it for the current head.
- If checks and review are both clean, evaluate merge gates.

Avoid repeated human updates for unchanged waiting states.

## Merge Gates

Never merge unless all are true:

- Required checks are green.
- No unresolved blocking review feedback remains.
- The branch has no unresolved merge conflicts.
- The latest head SHA has an explicit independent approval or no-blocking finding attached to the PR.
- The PR satisfies the linked issue acceptance criteria.
- No human decision is pending.

Never merge based only on absence of comments, an off-PR review, a stale review, or a review request.

## After Merge

After a PR merges or closes:

1. Run a short delivery retro.
2. Identify where time was lost: issue selection, worker start, first push, local gates, CI, review, blocker response, or merge.
3. Turn concrete lessons into workflow updates, prompts, checklists, batching criteria, polling cadence, or delegation patterns.
4. Then start the next backlog planning pass.

Do not let the repo sit idle after merge unless the backlog is exhausted or a real blocker exists.

## Backlog Planning Pass

Before starting new work:

1. Review recent merged PRs for unaddressed post-merge comments.
2. Create or confirm issues for any valid post-merge findings.
3. Review all open GitHub issues.
4. Classify candidates into same-batch, parallel-safe, or serial.
5. Select the highest-value same-batch PR first.
6. Start a second parallel lane only when it is clearly independent and the active lane has no actionable blocker.

Prefer one coherent PR for related issues when the combined diff remains reviewable.

## Batch Classification

Same-batch issues share at least one of:

- Subsystem.
- Safety boundary.
- Acceptance criteria.
- Fixtures or tests.
- Likely touched files.
- State or migration contracts.

Parallel-safe issues have:

- Different subsystems.
- Different files.
- Independent tests.
- Low rebase risk.
- No shared state contracts or migrations.

Serial issues are:

- Safety-critical.
- High-conflict.
- Review-sensitive.
- Dependent on an active PR.

If parallel lanes cause conflicts, stall reviews, or slow blocker response, collapse back to one active implementation lane.

## Status Updates

Keep routine updates concise and state-transition based.

Send updates when:

- PR opened.
- Blocker found and fix work started.
- Fix pushed.
- CI failed in a nontrivial way.
- Independent review is clean.
- PR merged.
- Backlog exhausted.
- A genuine blocker requires human input.
- A workflow adjustment was made after retro.

Avoid sending repeated updates that only say nothing changed. Do not paste raw logs, command output, stack traces, file listings, or test dumps into chat updates. Summarize the human-relevant result.

## Safety Rules

- GitHub state is authoritative.
- Read the issue, PR, review comments, and CI status before deciding.
- Do not duplicate work.
- Keep fixes scoped to the PR and blocker.
- Do not weaken safety guards to make tests pass.
- Do not expose credentials to development tooling.
- Do not merge failing required checks.
- Escalate product, architecture, safety, credential, spending, or unrecoverable blockers.

## Worker Prompt Template

Use this shape when delegating implementation or review-fix work:

```text
Repo: <owner/repo>
PR/Issue: <url>
Current head SHA: <sha>
Task: <implementation or review-fix goal>
Acceptance criteria: <bullets>
Blocking findings: <file/line/comment refs, if any>
Constraints: keep changes scoped, preserve safety guards, do not refactor unrelated code.
Before push: run <exact local gates>.
After push: report commit SHA, tests run, and any residual risk.
```

## Independent Review Prompt Template

```text
Review PR <url> at exact head SHA <sha>.
Linked issue: <url>
Acceptance criteria: <bullets>
Changed-file summary: <summary>
Safety checklist: <project-specific safety constraints>

Return blocking findings only. Include file and line references. Explicitly confirm whether the exact head SHA was reviewed. If no blockers are found, say that the exact head SHA has no blocking findings.
```
